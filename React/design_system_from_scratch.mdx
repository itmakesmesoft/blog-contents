AI 캐릭터 채팅앱의 크리에이터를 위한 스튜디오 서비스를 만들고 있다. 크리에이터가 캐릭터를 만들고, 스토리를 작성하고, 이미지와 음악을 생성하는 도구다.

스튜디오에는 같은 기능의 UI가 여러 페이지에 걸쳐 반복되었다. 버튼, 텍스트 입력, 드롭다운, 토스트 알림. 이들은 거의 모든 페이지에 존재했지만, 페이지마다 새로 구현되어 있었다. 같은 "저장" 버튼인데 어떤 페이지에서는 패딩이 `py-2`이고, 다른 페이지에서는 `py-3`이었다. 모서리 반경도 `rounded-lg`와 `rounded-xl`이 섞여 있었다. 텍스트 입력에 붙는 라벨, 에러 메시지, 글자수 카운터의 위치와 스타일도 구현자마다 달랐다.

디자인이 변경되면 문제가 드러났다. 디자이너가 "primary 버튼의 호버 색상을 바꿔주세요"라고 하면, 프로젝트 전체를 검색해서 해당 버튼을 사용하는 모든 곳을 찾아가며 수정해야 했다. 찾지 못한 곳은 이전 디자인 그대로 남았다. 같은 컴포넌트인데 페이지마다 미묘하게 달라지는 현상이 계속 누적되었다.

각 구현이 디자이너의 요구사항과 미묘하게 달랐다. 디자이너가 정의한 버튼은 4가지 variant, 4가지 color, 4가지 size의 조합이었다. 그런데 구현된 버튼들은 이 조합 중 일부만 따르고, 나머지는 임의로 스타일을 지정하고 있었다. 드롭다운의 키보드 내비게이션은 어떤 곳에서는 구현되어 있고, 어떤 곳에서는 빠져 있었다.

공용 컴포넌트를 체계적으로 만들기로 했다.

---

## 무엇을 공용화할 지 결정하기

프로젝트 전체를 돌아보며 반복적으로 사용되는 UI를 검토했다. 어떤 컴포넌트가 중복되고 있는지, 각각 어떤 변형이 존재하는지 정리했다.

버튼이 가장 많았다. 거의 모든 페이지에 존재했다. variant(fill, outline, text, ghost), color(primary, secondary, black, destructive), size(xs, sm, md, lg), corner(rounded, pill, square 등)의 조합이 페이지마다 제각각으로 구현되어 있었다. 어떤 곳은 `<button>`이고, 어떤 곳은 `<a>` 태그였고, 또 어떤 곳은 Next.js의 `<Link>`였다. loading 상태 처리도 통일되어 있지 않았다.

텍스트 입력도 문제였다. 라벨, 에러 메시지, 글자수 카운터가 각각 다르게 붙어 있었다. 어떤 곳은 라벨이 위에, 어떤 곳은 옆에 있었다. 글자수 카운터가 오른쪽 아래에 붙는 곳도 있고, 텍스트 영역 바깥에 별도로 표시되는 곳도 있었다. react-hook-form과 함께 쓰이는 곳(uncontrolled)과 실시간 글자수를 표시해야 하는 곳(controlled)이 혼재했다.

드롭다운은 옵션 선택 UI가 곳곳에 있었지만, 접근성 처리가 제각각이었다. 키보드로 옵션을 탐색할 수 있는 곳도 있고, 마우스로만 동작하는 곳도 있었다. 포커스 관리, ARIA 속성, 열림/닫힘 애니메이션이 구현마다 달랐다.

토스트 알림은 성공이나 에러 메시지를 표시하는 용도로 여러 페이지에서 사용되고 있었다. 어떤 곳은 로컬 state로 관리하고, 어떤 곳은 `window.alert`를 쓰고 있었다. 표시 위치, 지속 시간, 사라지는 방식이 모두 달랐다.

사이드바가 가장 복잡했다. 열림/닫힘 상태, 메뉴 구조, 중첩 메뉴, 활성 메뉴 표시, 접힌 상태에서의 아이콘 전용 표시 등 요구사항이 많았다. 메뉴 구조가 하드코딩되어 있어서 새로운 메뉴를 추가하려면 레이아웃 코드를 직접 수정해야 했다.

각 컴포넌트의 변형 패턴을 정리하고, 어떤 API가 필요한지 추렸다. 사용처를 먼저 분석하고, 거기서 공통 인터페이스를 뽑아내는 순서로 진행했다.

## API 설계하기

각 컴포넌트를 구현할 땐, 실제 사용처에서 어떤 패턴으로 주로 사용하는지를 점검했고, 이걸 cva를 이용해 조합 형태로 스타일을 추가했다. 내가 구현했던 디자인 시스템의 컴포넌트 중 몇 가지만 추려서 소개해보겠다.

### Button

사용처를 분석하면 버튼의 변형은 네 가지 축으로 나뉘었다. variant(fill, outline, text, ghost), color(primary, secondary, black, destructive), size(xs, sm, md, lg), corner(rounded, roundedXl, square, pill, roundedXs). 이 축들의 조합이 실제 스타일을 결정한다. variant가 `fill`이고 color가 `primary`이면 배경이 핑크, 텍스트가 흰색이다. variant가 `outline`이고 color가 `primary`이면 배경은 투명하고 테두리가 핑크다.

문제는 조합의 수가 많다는 것이었다. variant 4 x color 4 = 16가지 색상 조합. 여기에 size 4, corner 5까지 곱하면 경우의 수가 폭발한다. 이걸 조건문으로 관리하면 코드가 읽기 어려워진다.

`class-variance-authority`(CVA)의 `compoundVariants`를 선택했다. 각 축(variant, size, corner)은 독립적으로 클래스를 생성하고, variant와 color의 교차점만 `compoundVariants`로 정의한다. 16개의 교차 조합을 선언적으로 나열하면 끝이다. 새로운 variant를 추가할 때도 해당 축에 값을 추가하고, 필요한 교차 조합만 추가하면 된다.

```tsx
// 이전: 페이지마다 제각각
<button className="bg-[#FF2E7F] hover:bg-[#FF5A98] text-white px-4 py-2 rounded-lg font-semibold">
  저장
</button>

// 이후: variant 조합으로 통일
<Button variant="fill" color="primary" size="sm" corner="rounded">
  저장
</Button>
```

다형성(Polymorphic)도 필요했다. 사용처를 보면 같은 버튼 디자인이 `<button>`, `<a>`, Next.js `<Link>`로 렌더링되어야 하는 경우가 모두 존재했다. `as` prop을 받아서 렌더링 엘리먼트를 변경할 수 있도록 했다.

```tsx
// 일반 버튼
<Button variant="fill" color="primary">저장</Button>

// 링크지만 버튼 디자인
<Button as="a" href="/characters" variant="outline" color="secondary">
  캐릭터 목록
</Button>

// Next.js Link
<Button as={Link} href="/settings" variant="text" color="black">
  설정
</Button>
```

세 가지 모두 같은 variant 시스템을 공유하면서, 렌더링되는 HTML 엘리먼트만 다르도록 해야했다.

여기서 중요하게 생각한 건 타입 안전성(Type Safe)이었다. `as="a"`를 전달하면 `href`가 자동완성되어야 하고, `as="button"`이면 `href`를 전달할 때 타입 에러가 나길 원했다.

#### 유니온 타입 사용하기

처음에는 `as`의 타입을 단순하게 정의했다.

```tsx
type ButtonProps = {
  as?: "button" | "a" | typeof Link;
  variant?: "fill" | "outline" | "text";
  href?: string; // <a>를 위해 수동으로 추가
  target?: string; // <a>를 위해 수동으로 추가
  // ...
};
```

동작은 했다. 하지만 두 가지 문제가 드러났다.

첫째, `as="button"`일 때도 `href`를 전달할 수 있었다. 즉, 엘리먼트에 맞지 않는 prop을 전달해도 타입 에러가 나지 않았다.

```tsx
// TypeScript: OK (문제 없다고 판단)
// 런타임: <button href="/somewhere">이 렌더링됨 — 의미 없는 HTML
<Button as="button" href="/somewhere">
  저장
</Button>
```

둘째, 새로운 엘리먼트를 지원해야 할 때마다 해당 엘리먼트의 고유 props를 `ButtonProps`에 수동으로 추가해야 했다. 예를 들어 `as="label"`를 지원하려면 기존 방식으로는 `label` 태그애만 존재하는 `htmlFor`과 같은 props를 추가해야 했다. `as`에 올 수 있는 엘리먼트가 늘어날수록 `ButtonProps`가 비대해지고, 어떤 prop이 어떤 엘리먼트에 속하는지 구분이 되지 않게 된다.

#### 제네릭 타입 적용하기

난 `as`에 전달된 엘리먼트에 따라 허용되는 props가 자동으로 달라지는 것을 원했다.

[kciter의 다형성 컴포넌트 글](https://kciter.so/posts/polymorphic-react-component/)과 [정민의 Polymorphic Component 글](https://www.jeong-min.com/79-polymorphic-component/)을 참고했다. 아이디어는 `as`에 전달된 엘리먼트 타입을 제네릭 `T`로 받아서, `T`의 고유 props를 React의 타입 시스템에서 자동으로 추출하는 것이다.

```tsx
type AsProp<T extends React.ElementType> = {
  as?: T;
};

type PolymorphicComponentProps<
  T extends React.ElementType,
  Props = Record<string, never>,
> = AsProp<T> &
  Props &
  Omit<React.ComponentPropsWithoutRef<T>, keyof AsProp<T> | keyof Props> & {
    ref?: PolymorphicRef<T>;
  };
```

풀어 쓰면 이렇다.

`React.ComponentPropsWithoutRef<T>`는 React가 제공하는 유틸리티 타입이다. `T`에 `"a"`를 넣으면 `href`, `target`, `rel` 같은 `<a>` 엘리먼트의 모든 props가 나오고, `"button"`을 넣으면 `<button>` 엘리먼트의 props가 나온다.

`Omit`은 충돌을 방지한다. `<button>` 엘리먼트에도 `disabled`가 있고, Button 자체 props에도 `disabled`가 있다. `Omit<React.ComponentPropsWithoutRef<T>, keyof Props>`로 Button 자체 props와 이름이 겹치는 것을 제거하여, Button의 `disabled`가 우선하도록 한다.

사용처에서 보면 이렇게 된다.

```tsx
// as="a" → T가 "a"로 추론 → href, target이 허용됨
<Button as="a" href="/characters">캐릭터</Button>  // OK

// as="button" → T가 "button"으로 추론 → href가 존재하지 않음
<Button as="button" href="/characters">캐릭터</Button>  // 타입 에러!

// as={Link} → T가 typeof Link로 추론 → prefetch, replace가 허용됨
<Button as={Link} href="/settings" prefetch={false}>설정</Button>  // OK
```

첫 번째 시도에서 발생했던 두 가지 문제가 모두 해결됐다. `as`에 따라 허용되는 props가 자동으로 달라지고, 잘못된 props를 전달하면 타입 에러가 난다. 새로운 엘리먼트를 지원해도 `ButtonProps`를 수정할 필요가 없다.

여기까지는 참고 글을 따라가면 됐다. 문제는 그 다음이었다.

#### `forwardRef`와의 충돌 해결하기

Button은 외부에서 ref를 받아야 하는 경우가 있었다. Tooltip의 트리거로 Button을 사용하거나, 포커스를 이동시켜야 할 때 ref가 필요하다. 그래서 `forwardRef`를 사용해야 했다.

문제는 `forwardRef`의 타입 시그니처가 제네릭 컴포넌트를 지원하지 않는다는 것이었다. `forwardRef`는 두 개의 타입 매개변수를 받는다. ref의 타입과 props의 타입이다.

```tsx
// forwardRef의 타입 시그니처 (단순화)
function forwardRef<T, P>(
  render: (props: P, ref: Ref<T>) => ReactNode,
): ForwardRefExoticComponent<PropsWithoutRef<P> & RefAttributes<T>>;
```

여기서 `T`와 `P`는 호출 시점에 고정된다. `forwardRef<HTMLButtonElement, ButtonProps>`로 쓰면 ref는 항상 `HTMLButtonElement`이고, props는 항상 `ButtonProps`다. 우리가 만든 `ButtonProps<T>`의 제네릭 `T`가 들어갈 자리가 없다.

```tsx
// 이렇게 쓰면 T가 "button"으로 고정되어 버린다
const Button = React.forwardRef<HTMLButtonElement, ButtonProps<"button">>((props, ref) => {
  const Component = props.as || "button"
  return <Component ref={ref} {...restProps} />
})

// 사용처에서 as="a"를 전달해도, ref는 HTMLButtonElement 타입
const linkRef = useRef<HTMLAnchorElement>(null)
<Button as="a" ref={linkRef} />  // 타입 에러: HTMLAnchorElement는 HTMLButtonElement에 할당 불가
```

`as="a"`를 전달했으니 ref의 타입도 `HTMLAnchorElement`가 되어야 하는데, `forwardRef`가 `HTMLButtonElement`로 고정해 버린다. 앞에서 만든 제네릭 타입이 `forwardRef`를 통과하면서 소거되는 것이다.

`forwardRef`를 감싸는 제네릭 래퍼 함수를 만들어 보기도 했다.

```tsx
function polymorphicForwardRef<T extends React.ElementType, P>(
  render: (props: P, ref: PolymorphicRef<T>) => ReactNode,
) {
  return React.forwardRef(render) as any; // 결국 any로 도피
}
```

하지만 타입 단언 `as any`를 쓰는 순간 타입 안전성이 무너진다. 래퍼 함수 내부에서 타입을 정확하게 연결하려 하면 TypeScript의 타입 추론 한계에 부딪혀서 에러가 발생했다.

결국 방향을 바꿨다. `forwardRef`의 내부 타입을 고치는 걸 포기하고, 외부에 노출되는 타입을 별도로 선언했다.

```tsx
// 1. 외부에 노출할 컴포넌트 타입을 먼저 선언한다
// 이 타입은 제네릭 T를 받는다
type ButtonComponent = <T extends React.ElementType = "button">(
  props: ButtonProps<T>,
) => React.ReactNode;

// 2. forwardRef로 구현하되, 변수의 타입을 ButtonComponent로 지정한다
export const Button: ButtonComponent = React.forwardRef(
  <T extends React.ElementType = "button">(
    props: ButtonProps<T>,
    ref: PolymorphicRef<T>,
  ) => {
    const Component = props.as || "button";
    return <Component ref={ref} {...restProps} />;
  },
);
```

원리는 이렇다. `React.forwardRef`의 반환 타입은 `ForwardRefExoticComponent<...>`인데, 이 타입은 제네릭 `T`를 포함하지 않는다. 하지만 `const Button: ButtonComponent = ...`로 선언하면, TypeScript는 외부에서 `Button`을 사용할 때 `ButtonComponent` 타입을 기준으로 타입 체크를 한다. `ButtonComponent`는 제네릭 `T`를 받기 때문에, `as`에 전달된 값에 따라 `T`가 추론된다.

`forwardRef` 내부에서는 `T`가 소거되지만, 외부에서 보는 타입은 `ButtonComponent`이므로 제네릭이 살아 있다. `forwardRef`의 타입 한계를 우회하면서, 사용처에서는 타입 안전성이 유지된다.

```tsx
const buttonRef = useRef<HTMLButtonElement>(null)
const anchorRef = useRef<HTMLAnchorElement>(null)

<Button ref={buttonRef}>저장</Button>                    // OK
<Button as="a" ref={anchorRef} href="/">홈</Button>      // OK
<Button as="a" ref={buttonRef} href="/">홈</Button>      // 타입 에러: HTMLButtonElement ≠ HTMLAnchorElement
<Button as="button" href="/">홈</Button>                  // 타입 에러: href는 button에 없음
```

loading 상태도 설계 판단이 필요했다. 로딩 중일 때 Spinner를 보여줘야 하는데, Spinner가 버튼의 크기를 변경하면 옆에 있는 요소들이 밀린다. Spinner를 `absolute` 포지션으로 버튼 위에 오버레이하고, `bg-inherit`로 배경색을 그대로 상속받아 기존 텍스트를 가리는 방식을 선택했다. 버튼의 크기는 변하지 않고, Spinner만 기존 콘텐츠 위에 표시된다.

### Textarea

react-hook-form을 사용하는 곳에서는 uncontrolled 방식(`defaultValue` + `ref`)이 필요하고, 실시간 글자수를 표시하는 곳에서는 controlled 방식(`value` + `onChange`)이 필요했다. 두 가지 모드를 별도의 컴포넌트로 만들면 API가 분산된다. 하나의 컴포넌트에서 `value` prop의 유무로 모드를 판별하도록 했다. `value`가 전달되면 controlled, 그렇지 않으면 uncontrolled로 동작한다.

부가 UI의 배치가 더 큰 문제였다. 라벨은 위에, 에러 메시지는 아래에, 글자수 카운터는 오른쪽 아래에, 어떤 곳에서는 왼쪽에 아이콘이 붙기도 했다. 이 조합을 각각 prop으로 만들면(`hasLabel`, `hasCounter`, `iconPosition` 등) 조합이 폭발한다.

4방향 Affix 시스템을 도입했다. `topAffix`, `bottomAffix`, `leftAffix`, `rightAffix` 네 개의 슬롯을 열어두고, 사용처에서 원하는 UI를 원하는 위치에 넣도록 했다. Textarea는 자신이 어떤 부가 UI가 붙는지 알 필요가 없다. 슬롯에 무엇이 들어오든 레이아웃만 보장한다.

```tsx
// 라벨 + 에러 + 글자수 카운터
<Textarea
  label="스토리 본문"
  error={errors.content?.message}
  showCount
  maxLength={5000}
  topAffix={<Badge>AI 추천</Badge>}
  bottomAffix={<HelpText>마크다운 문법을 지원합니다</HelpText>}
/>

// 아이콘 + 단순 입력
<Textarea
  leftAffix={<SearchIcon />}
  placeholder="검색어를 입력하세요"
/>

// 부가 UI 없이 bare textarea
<Textarea placeholder="메모" />
```

Affix가 하나라도 있으면 wrapper `div`로 감싸서 border와 focus 스타일을 wrapper에 적용한다. Affix가 하나도 없으면 bare `textarea`를 렌더링하여 불필요한 DOM 래핑을 피했다.

### Dropdown

드롭다운에서 어려운 건 인터랙션이다. 키보드 탐색(Arrow Up/Down, Enter, Escape), 포커스 관리, 외부 클릭 시 닫기, ARIA 속성, 스크린 리더 대응. 이 모든 것을 직접 구현하면 코드가 복잡해지고, 엣지 케이스를 놓치기 쉽다.

Radix UI를 기반으로 선택했다. Radix가 접근성과 키보드 인터랙션을 처리한다. 우리는 프로젝트에 맞는 디자인 인터페이스만 정의했다. size(xs, sm, md, lg), cornerRadius(sm, lg), footer 슬롯, 그리고 동적 z-index 연동(`useZIndex`).

Compound Component 패턴으로 `Dropdown.Root`, `Dropdown.Trigger`, `Dropdown.Content`, `Dropdown.Item`, `Dropdown.Footer`를 나누었다. 동시에, 대부분의 사용처에서는 단순한 옵션 선택만 필요했기 때문에 `DropdownMenu`라는 편의 컴포넌트도 함께 제공했다.

```tsx
// 편의 컴포넌트: 대부분의 사용처
<DropdownMenu
  trigger="정렬 기준"
  options={[
    { label: "최신순", value: "latest" },
    { label: "인기순", value: "popular" },
  ]}
  value={sortBy}
  onValueChange={setSortBy}
  size="sm"
  footer={<Button variant="text">초기화</Button>}
/>

// Compound Component: 세밀한 제어가 필요한 경우
<Dropdown.Root>
  <Dropdown.Trigger>
    <Button variant="outline">옵션</Button>
  </Dropdown.Trigger>
  <Dropdown.Content>
    <Dropdown.Item onSelect={handleEdit}>수정</Dropdown.Item>
    <Dropdown.Item onSelect={handleDelete} className="text-red-500">삭제</Dropdown.Item>
    <Dropdown.Footer>
      <span className="text-xs text-gray-400">Shift+Click으로 바로 삭제</span>
    </Dropdown.Footer>
  </Dropdown.Content>
</Dropdown.Root>
```

접근성 인터랙션은 Radix에 맡기고, 우리는 프로젝트에 맞는 디자인 API만 정의했다.

### Sidebar

사이드바는 요구사항이 가장 많은 컴포넌트였다. 열림/닫힘 상태 관리, 너비 전환 애니메이션, 메뉴 구조, 중첩 메뉴, 활성 메뉴 표시, 접힌 상태에서의 아이콘 전용 표시까지.

Compound Component 패턴을 선택했다. `Sidebar.Provider`, `Sidebar.Sidebar`, `Sidebar.Inset`, `Sidebar.Scrollable`, `Sidebar.Menu`, `Sidebar.GroupMenu`, `Sidebar.MenuItem` 7개의 서브 컴포넌트를 하나의 네임스페이스로 묶었다. 각 서브 컴포넌트는 독립적으로 관심사를 가지고, Context를 통해 상태를 공유한다.

```tsx
// 레이아웃에서의 사용
<Sidebar.Provider>
  <Sidebar.Sidebar topAffix={<Logo />}>
    <Sidebar.Scrollable>
      <Sidebar.GroupMenu title={<Sidebar.MenuItem>캐릭터</Sidebar.MenuItem>}>
        <Sidebar.Menu href="/characters" active={isActive("/characters")}>
          <Sidebar.MenuItem leftAddon={<ListIcon />}>목록</Sidebar.MenuItem>
        </Sidebar.Menu>
        <Sidebar.Menu href="/characters/new">
          <Sidebar.MenuItem leftAddon={<PlusIcon />}>
            새 캐릭터
          </Sidebar.MenuItem>
        </Sidebar.Menu>
      </Sidebar.GroupMenu>
      <Sidebar.GroupMenu title={<Sidebar.MenuItem>스토리</Sidebar.MenuItem>}>
        {/* 중첩 메뉴도 같은 구조 */}
      </Sidebar.GroupMenu>
    </Sidebar.Scrollable>
  </Sidebar.Sidebar>
  <Sidebar.Inset>{children}</Sidebar.Inset>
</Sidebar.Provider>
```

새 메뉴를 추가하려면 `<Sidebar.Menu>`를 하나 더 넣으면 된다. 레이아웃 코드를 수정할 필요가 없다.

스타일 전파에는 CSS Variables를 사용했다. Provider가 `--sidebar-open-width`, `--sidebar-closed-width`, `--sidebar-transition-duration`, `--sidebar-menu-active-text` 같은 변수를 루트에 설정한다. 하위 컴포넌트들은 이 변수를 참조해서 스타일을 결정한다. 너비, 전환 속도, 색상을 props로 전달하지 않아도 CSS Variables가 전파한다.

중첩 메뉴의 인덴트 처리가 까다로운 부분이었다. 메뉴가 1단, 2단, 3단으로 중첩될 때 각 단계마다 인덴트가 증가해야 한다. 깊이를 props로 전달하면 사용처에서 깊이를 직접 관리해야 한다. CSS Variables의 누적으로 해결했다. `SidebarGroupMenu`가 열릴 때 `--current-indent: calc(var(--sidebar-menu-indent, 0) + 1)`을 설정한다. 자식 메뉴는 이 값을 `--sidebar-menu-indent`로 읽고, 다시 +1하여 자신의 자식에게 전달한다. 깊이를 명시적으로 전달할 필요 없이 CSS Variables가 자동으로 누적된다.

## 서버에서도 Sidebar의 열림 상태를 알도록 하기

앞의 네 컴포넌트(Button, Textarea, Dropdown, Toast)는 클라이언트에서만 동작하면 됐다. 버튼이 서버에서 렌더링될 때 variant가 틀려도 하이드레이션 후 즉시 교정되고, 사용자는 눈치채지 못한다. 크기가 변하지 않기 때문이다.

사이드바는 달랐다.

### Layout Shift

Next.js App Router에서 사이드바는 레이아웃의 일부다. 서버 렌더링 시점에 사이드바의 열림/닫힘 상태를 모르면 문제가 발생한다.

서버는 기본값(열림)으로 렌더링한다. 사이드바 너비 244px. 콘텐츠 영역은 나머지를 차지한다. HTML이 브라우저에 도착하고, JavaScript가 실행되어 하이드레이션이 완료된다. 이때 클라이언트가 "이 사용자는 사이드바를 접어놨다"는 것을 알게 된다. 사이드바가 92px로 좁아진다. 콘텐츠 영역이 갑자기 넓어진다.

사용자에게는 페이지가 로드된 후 레이아웃이 "점프"하는 것으로 보인다. Layout Shift다.

### 쿠키로 서버와 클라이언트를 연결하기

아이디어는 단순하다. 서버에서도 사이드바 상태를 알 수 있으면 된다. 서버와 클라이언트 양쪽에서 접근 가능한 저장소는 쿠키다.

Provider를 `async` Server Component로 만들었다. Next.js의 `cookies()` API로 서버에서 쿠키를 읽고, 그 값을 클라이언트 Provider의 초기값으로 전달한다.

```tsx
// Provider.tsx — async Server Component
"use server";

import { cookies } from "next/headers";
import { InternalSidebarProvider } from "./context";

export const SidebarProvider = async ({
  children,
  openWidth = "244px",
  closedWidth = "92px",
  defaultOpen,
}: {
  children: React.ReactNode;
  openWidth?: string;
  closedWidth?: string;
  defaultOpen?: boolean;
}) => {
  const cookieStore = await cookies();
  const sidebarOpen =
    defaultOpen ?? cookieStore.get("sidebar-open")?.value === "true";

  return (
    <InternalSidebarProvider
      defaultOpen={sidebarOpen}
      style={{
        "--sidebar-transition-duration": "300ms",
        "--sidebar-open-width": openWidth,
        "--sidebar-closed-width": closedWidth,
        // ...
      }}
    >
      {children}
    </InternalSidebarProvider>
  );
};
```

`SidebarProvider`는 `async` 함수다. `await cookies()`로 쿠키 저장소에 접근하고, `sidebar-open` 쿠키의 값을 읽는다. `defaultOpen` prop이 명시적으로 전달되면 그 값을 우선하고, 그렇지 않으면 쿠키 값을 사용한다. 이 값이 클라이언트 Provider(`InternalSidebarProvider`)의 `defaultOpen`으로 전달된다.

클라이언트 쪽에서는 상태가 변경될 때마다 쿠키를 업데이트한다.

```tsx
// context.tsx — 클라이언트 Provider
"use client";

import { setCookie } from "./client-cookie";

export const InternalSidebarProvider = ({
  children,
  defaultOpen = false,
  style,
  ...restProps
}: {
  children: React.ReactNode;
  defaultOpen?: boolean;
} & HTMLAttributes<HTMLDivElement>) => {
  const [internalOpen, setInternalOpen] = useState<boolean>(defaultOpen);

  const setOpen = (newOpen: boolean) => {
    setInternalOpen(newOpen);
    setCookie("sidebar-open", String(newOpen), 30); // 30일 유지
  };

  return (
    <sidebarContext.Provider value={{ open: internalOpen, setOpen }}>
      <div
        style={{
          ...style,
          "--sidebar-width": internalOpen
            ? "var(--sidebar-open-width)"
            : "var(--sidebar-closed-width)",
        }}
      >
        {children}
      </div>
    </sidebarContext.Provider>
  );
};
```

`setOpen`이 호출될 때마다 React 상태와 쿠키를 동시에 업데이트한다. 쿠키의 만료 기간은 30일이다. 다음 요청에서 서버가 이 쿠키를 읽고, 처음부터 올바른 상태로 렌더링한다.

흐름을 정리하면 이렇다. 사용자가 사이드바를 접는다. 클라이언트가 `setCookie("sidebar-open", "false", 30)`을 실행한다. 다음에 페이지를 열면, 서버가 `cookies().get("sidebar-open")`으로 `"false"`를 읽는다. 사이드바를 접힌 상태(92px)로 서버 렌더링한다. 브라우저에 도착한 HTML은 처음부터 접힌 상태다. 하이드레이션 후에도 상태가 동일하므로 Layout Shift가 발생하지 않는다.

## 마무리

디자인 시스템을 만드는 과정은 결코 만만치 않았다. 특히 SSR 환경에서 레이아웃 시프트를 잡으면서, 사용자 경험이라는 게 결국 이런 사소한 디테일에 있다는 걸 느꼈다.

이러한 과정을 거치고 뼈대를 만들어 가다보니 생산성이 매우 향상되었다. 새 페이지를 만들 때 버튼 패딩이나 드롭다운 키보드 내비게이션을 고민하지 않고, 컴포넌트를 조합하면 되니까, 서비스 기능 자체에 더 고민할 수 있게 됐다.

디자인 시스템은 한 번 만들고 끝나는 게 아니다. 서비스가 커지면 새로운 요구사항이 나오고, 그때마다 시스템도 같이 고쳐야 한다. 이번에 기반을 깔아둔 게 그때 도움이 될 거라고 생각한다.
