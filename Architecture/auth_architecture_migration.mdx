운영 중인 서비스에서 가장 건드리기 두려운 영역을 꼽으라면, 인증이다.

사용자가 30분마다 로그인이 풀린다는 제보가 들어왔다. 원인은 허무하게도 리프레시 토큰과 액세스 토큰의 만료 시간이 둘 다 30분으로 설정되어 있던 것이었다. 그런데 이 값만 바꿔서는 안 됐다.

우리 서비스는 NestJS 기반 OAuth API(OpenAPI)에서 토큰을 발급하고, 그 uid로 Supabase 사용자 데이터에 접근하는 구조다. 토큰이 DB와 localStorage에 동시에 저장되고 있었고, 어떤 로직은 DB에서, 어떤 로직은 localStorage에서 읽었다. 갱신 로직도 클라이언트와 서버에 제각각이었다. 하나를 고치면 다른 곳에서 터지는 구조.

만료 시간을 고치는 건 미봉책이었다. 토큰 저장소를 쿠키로 일원화하고, 갱신을 미들웨어 한 곳에서 처리하기로 했다.

---

## 기존 구조의 문제

```
클라이언트 → NestJS OAuth API(OpenAPI) → OAuth Provider (Kakao/Google/...)
              ↓
           토큰 발급 → 토큰 내 uid로 Supabase(사용자 데이터) 접근
           - 토큰이 DB와 localStorage에 동시에 저장
           - 어떤 로직은 DB에서, 어떤 로직은 localStorage에서 토큰을 읽음
```

사용자에게 직접 영향을 주는 문제가 세 가지 있었다.

서비스 사용 중 간헐적으로 세션이 끊겼다. 리프레시 토큰과 액세스 토큰의 만료 시간이 모두 30분이니 당연한 결과다.

`getCurrentUser()`를 호출할 때마다 내부적으로 리프레시 토큰으로 액세스 토큰을 갱신하는 구조였다. 토큰이 유효하든 아니든, 매 페이지 전환마다 갱신 요청이 나갔다.

DB와 localStorage 사이의 토큰 상태가 어긋나는 경우도 있었다. 두 저장소가 달라지면 만료된 토큰으로 인증을 시도하게 되고, 사용자가 갑자기 로그인 화면으로 튕겼다.

토큰을 쓰는 이유는 토큰만으로 인증을 관리하기 위해서인데, 그 토큰을 DB에 저장하고 다시 조회하는 구조 자체가 잘못되어 있었다. 토큰 조회를 위한 별도 API 요청, localStorage와의 동기화 로직. 이것들이 남아 있는 한 같은 류의 버그는 계속 나올 수밖에 없다.

## 쿠키 일원화

### 쿠키 설계

DB와 localStorage에 흩어져 있던 토큰을 쿠키로 통일했다.

```
클라이언트 → OpenAPI
              ↓
           토큰을 쿠키에서만 관리
           - Access Token  → non-httpOnly 쿠키
           - Refresh Token → httpOnly 쿠키
           미들웨어가 자동 갱신
```

두 종류의 토큰을 의도적으로 다르게 설정했다.

```
Access Token  → non-httpOnly, maxAge: 1시간
Refresh Token → httpOnly,     maxAge: 30일
```

Access Token을 non-httpOnly로 설정한 건 의도적인 트레이드오프다. 클라이언트에서 외부 API에 `Authorization: Bearer` 헤더로 직접 전달해야 하므로 JavaScript 접근이 필요했다. 만료 시간을 1시간으로 짧게 잡아 탈취 시 피해를 제한했고, Refresh Token은 `httpOnly` + `secure`로 XSS를 차단했다.

### 미들웨어에서 토큰 자동 갱신

쿠키 기반으로 전환하면서 가장 큰 이점은 미들웨어에서 갱신을 처리할 수 있게 된 것이다. Next.js 미들웨어는 모든 요청 전에 실행되므로, 여기서 만료를 확인하고 갱신하면 이후 Server Component든 API Route든 항상 유효한 토큰을 쓸 수 있다.

```typescript
// middleware.ts
const refreshAccessToken = async (request: NextRequest) => {
  const accessToken = request.cookies.get(
    OAUTH_COOKIE_KEYS.ACCESS_TOKEN,
  )?.value;
  const refreshToken = request.cookies.get(
    OAUTH_COOKIE_KEYS.REFRESH_TOKEN,
  )?.value;

  if ((!accessToken || isTokenExpired(accessToken)) && refreshToken) {
    const tokenUrl = new URL(
      LOVEYDOVEY_API_ENDPOINTS.TOKEN,
      LOVEYDOVEY_OAUTH_CONFIG.BASE_URL,
    );
    const response = await fetch(tokenUrl.toString(), {
      method: "POST",
      headers: { "Content-Type": "application/x-www-form-urlencoded" },
      body: new URLSearchParams({
        grant_type: "refresh_token",
        client_id: LOVEYDOVEY_OAUTH_CONFIG.CLIENT_ID,
        client_secret: LOVEYDOVEY_OAUTH_CONFIG.CLIENT_SECRET,
        refresh_token: refreshToken,
      }),
    });

    if (!response.ok) return null;

    const tokenData = await response.json();
    const nextResponse = NextResponse.next();
    nextResponse.cookies.set(
      OAUTH_COOKIE_KEYS.ACCESS_TOKEN,
      tokenData.access_token,
      {
        path: "/",
        maxAge: TOKEN_EXPIRES.ACCESS_TOKEN_SECONDS,
        sameSite: "lax",
        httpOnly: false,
      },
    );
    if (tokenData.refresh_token) {
      nextResponse.cookies.set(
        OAUTH_COOKIE_KEYS.REFRESH_TOKEN,
        tokenData.refresh_token,
        {
          path: "/",
          maxAge: TOKEN_EXPIRES.REFRESH_TOKEN_SECONDS,
          sameSite: "lax",
          httpOnly: true,
          secure: process.env.NODE_ENV === "production",
        },
      );
    }
    return nextResponse;
  }
  return null;
};
```

클라이언트에서만 갱신하면 Server Component에서 만료된 토큰으로 데이터를 가져오게 된다. API Route에서만 갱신하면 API 호출 없는 페이지 네비게이션에서는 갱신이 안 된다. 미들웨어는 이 두 문제를 동시에 해결한다. 토큰이 유효하면 그냥 통과하고, 만료됐을 때만 OAuth 서버에 갱신 요청을 보낸다.

## 레거시 정리

새로운 구조를 만드는 것보다 기존 구조의 흔적을 제거하는 게 실제로 더 오래 걸렸다.

### localStorage 로직 제거

프로젝트 전반에 토큰을 저장하고 읽는 코드가 흩어져 있었다. 토큰을 다루는 유틸 함수가 여러 벌 있었고, 어떤 컴포넌트는 유틸을 쓰고 어떤 컴포넌트는 `localStorage.getItem()`을 직접 호출했다. 쿠키로 전환했는데 이런 코드가 남아 있으면, 쿠키에는 새 토큰이 있고 localStorage에는 옛 토큰이 남아 있는 상태가 된다. 어떤 요청은 유효한 토큰으로, 어떤 요청은 만료된 토큰으로 나가는 거다.

프로젝트 전체에서 `localStorage`를 참조하는 코드를 검색해서, 토큰 관련 읽기·쓰기를 쿠키 기반으로 교체하거나 제거했다. 중복된 유틸 함수도 하나로 합쳤다.

### 분산된 인증 검사

쿠키에 토큰이 멀쩡히 있는데 로그인 페이지로 튕기는 문제도 있었다. 미들웨어 외에도 개별 페이지와 컴포넌트에서 토큰을 직접 검사하고 리다이렉트하는 코드가 곳곳에 남아 있었기 때문이다.

예를 들어 어떤 페이지는 `useEffect`에서 localStorage의 토큰을 확인하고, 없으면 로그인으로 보냈다. 쿠키에는 유효한 토큰이 있지만 localStorage는 비어 있으니, 로그인된 사용자가 매번 튕겨나갔다.

이런 인증 검사 로직을 모두 찾아서 제거했다. 인증이 필요한 페이지의 접근 제어는 미들웨어가 담당하고, 컴포넌트는 미들웨어를 통과했으면 인증된 상태라고 신뢰하는 구조로 바꿨다.

## 회고

개발 2일, 사내 테스트 포함해서 1주일이 걸렸다. 전환 후 로그인 실패 리포트는 0건이다.

다만 미들웨어 갱신 방식에 잠재적인 한계는 있다. 동시에 여러 요청이 들어오면 각 미들웨어가 독립적으로 refresh token 갱신을 시도하면서 race condition이 생길 수 있다. OAuth 서버 호출 실패 시 폴백도 따로 구현하지 않았다. 현재는 갱신 실패 시 만료된 토큰이 그대로 전달되고, 이후 인증 로직에서 실패를 처리한다.

지금 서비스 규모에서는 문제가 발생하지 않았지만, 트래픽이 늘면 대응이 필요한 영역이다.
