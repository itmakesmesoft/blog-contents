운영 중인 서비스에서 가장 건드리기 두려운 영역을 꼽으라면, 인증이다.

사용자가 **30분마다 로그인이 풀리는** 문제가 보고되었다. 원인을 추적해 보니, 리프레시 토큰과 액세스 토큰의 만료 시간이 모두 30분으로 잘못 설정된 것이었다. 그런데 이 버그를 고치는 것만으로는 충분하지 않았다. 우리 서비스는 NestJS 기반 OAuth API(OpenAPI)에서 토큰을 발급하고, 그 토큰의 uid로 Supabase의 사용자 데이터에 접근하는 구조였다. 토큰 갱신 로직은 클라이언트와 서버에 걸쳐 제각각 구현되어 있었다. 하나를 고치면 다른 곳에서 문제가 터질 수 있는 구조였다.

버그를 고치는 것이 아니라, 같은 류의 버그가 재발하지 않도록 **구조 자체를 단순화**해야 했다. 토큰 저장소를 쿠키로 일원화하고, OpenAPI 서버만을 사용하여 토큰을 발급·갱신하는 구조로 전환하기로 했다.

---

## 기존 구조의 문제

#### Before

```
클라이언트 → NestJS OAuth API(OpenAPI) → OAuth Provider (Kakao/Google/...)
              ↓
           토큰 발급 → 토큰 내 uid로 Supabase(사용자 데이터) 접근
           - 토큰이 DB와 localStorage에 동시에 저장
           - 어떤 로직은 DB에서, 어떤 로직은 localStorage에서 토큰을 읽음
```

> 같은 토큰이 DB와 localStorage에 동시에 저장되고, 어떤 로직은 DB에서, 어떤 로직은 localStorage에서 토큰을 읽었다.

#### After

```
클라이언트 → OpenAPI
              ↓
           토큰을 쿠키에서만 관리
           - Access Token  → non-httpOnly 쿠키
           - Refresh Token → httpOnly 쿠키
           미들웨어가 자동 갱신
```

> 토큰 저장소가 쿠키 한 곳으로 통일되고, 미들웨어가 만료 시 자동으로 갱신한다.

사용자에게 직접 영향을 주는 문제가 세 가지 있었다.

**첫째, 서비스를 사용하는 도중에 간헐적으로 토큰이 만료되는 경우가 있었다.** 토큰 갱신로직이 잘못 설정되어 있었다. (리프레시 토큰과 엑세스 토큰의 만료시간이 모두 30분....)

**둘째, 매 페이지 전환마다 불필요한 토큰 갱신 요청이 발생했다.** 모든 컴포넌트에서 `getCurrentUser()`를 호출할 때마다 내부적으로 리프레시 토큰으로 액세스 토큰을 갱신하는 구조였다. 토큰이 유효하든 아니든, 호출할 때마다 갱신이 일어났다.

**셋째, DB와 localStorage 사이의 토큰 상태 불일치가 발생했다.** 같은 토큰이 데이터베이스와 localStorage에 각각 저장되어 있었는데, 어떤 로직은 DB에서, 어떤 로직은 localStorage에서 토큰을 읽었다. 두 저장소의 상태가 어긋나면 만료된 토큰으로 인증을 시도하게 되어, 사용자가 갑자기 로그인 화면으로 튕기는 문제가 발생했다.

이 버그들을 하나씩 패치하는 것은 미봉책이었다. 토큰을 사용하는 이유는 토큰만으로 인증을 관리하기 위함인데, 그 토큰을 DB에 저장하고 다시 조회하는 구조 자체가 잘못되어 있었다. 토큰 조회를 위한 별도 API 요청, localStorage와의 동기화 로직 — 이런 것들이 존재하는 한 같은 류의 버그는 계속 재발할 수밖에 없었다.

## 구현 — 쿠키 일원화

### 쿠키 설계

DB와 localStorage에 흩어져 있던 토큰을 쿠키로 통일했다. 두 종류의 토큰을 의도적으로 다르게 설정했다.

```
Access Token  → non-httpOnly, maxAge: 1시간
Refresh Token → httpOnly,     maxAge: 30일
```

Access Token을 non-httpOnly로 설정한 것은 **의도적인 보안 트레이드오프**다. 클라이언트에서 외부 API에 `Authorization: Bearer` 헤더로 직접 전달해야 하므로 JavaScript 접근이 필요했다. 대신 만료 시간을 1시간으로 짧게 설정하여, 탈취되더라도 피해를 제한했다. Refresh Token은 `httpOnly` + `secure`로 XSS 공격으로부터 보호했다.

이 설계로 세 가지 문제가 모두 해결된다.

- **30분 만료 문제**: Access Token 1시간, Refresh Token 30일로 적절한 만료 시간 설정
- **getCurrentUser() 비효율**: 토큰 갱신 로직이 미들웨어로 이동하여, `getCurrentUser()`가 더 이상 갱신을 책임지지 않는다
- **DB/localStorage 불일치**: 저장소가 쿠키 한 곳으로 통일되어 불일치가 원천적으로 불가능

### 미들웨어에서 토큰 자동 갱신

쿠키 기반 토큰 관리의 가장 큰 이점은 **미들웨어에서 토큰 갱신을 자동으로 처리**할 수 있다는 것이다. Next.js 미들웨어는 모든 요청에 선행하여 실행되므로, 여기서 토큰 만료를 확인하고 갱신하면 이후 실행되는 Server Component, API Route 어디서든 항상 유효한 토큰을 사용할 수 있다.

```typescript
// middleware.ts
const refreshAccessToken = async (request: NextRequest) => {
  const accessToken = request.cookies.get(
    OAUTH_COOKIE_KEYS.ACCESS_TOKEN,
  )?.value;
  const refreshToken = request.cookies.get(
    OAUTH_COOKIE_KEYS.REFRESH_TOKEN,
  )?.value;

  // 액세스 토큰이 만료되었고 리프레시 토큰이 있으면
  if ((!accessToken || isTokenExpired(accessToken)) && refreshToken) {
    const tokenUrl = new URL(
      LOVEYDOVEY_API_ENDPOINTS.TOKEN,
      LOVEYDOVEY_OAUTH_CONFIG.BASE_URL,
    );
    const response = await fetch(tokenUrl.toString(), {
      method: "POST",
      headers: { "Content-Type": "application/x-www-form-urlencoded" },
      body: new URLSearchParams({
        grant_type: "refresh_token",
        client_id: LOVEYDOVEY_OAUTH_CONFIG.CLIENT_ID,
        client_secret: LOVEYDOVEY_OAUTH_CONFIG.CLIENT_SECRET,
        refresh_token: refreshToken,
      }),
    });

    if (!response.ok) return null;

    const tokenData = await response.json();
    const nextResponse = NextResponse.next();
    nextResponse.cookies.set(
      OAUTH_COOKIE_KEYS.ACCESS_TOKEN,
      tokenData.access_token,
      {
        path: "/",
        maxAge: TOKEN_EXPIRES.ACCESS_TOKEN_SECONDS,
        sameSite: "lax",
        httpOnly: false,
      },
    );
    if (tokenData.refresh_token) {
      nextResponse.cookies.set(
        OAUTH_COOKIE_KEYS.REFRESH_TOKEN,
        tokenData.refresh_token,
        {
          path: "/",
          maxAge: TOKEN_EXPIRES.REFRESH_TOKEN_SECONDS,
          sameSite: "lax",
          httpOnly: true,
          secure: process.env.NODE_ENV === "production",
        },
      );
    }
    return nextResponse;
  }
  return null;
};
```

클라이언트에서만 갱신하면 Server Component에서 만료된 토큰으로 데이터를 가져오는 문제가 발생한다. API Route에서만 갱신하면 API 호출이 없는 페이지 네비게이션에서는 토큰이 갱신되지 않는다. 미들웨어 방식은 이 두 가지 한계를 모두 해결한다. 토큰이 유효한 동안에는 외부 호출 없이 통과하고, 만료된 경우에만 OAuth 서버에 갱신 요청을 보낸다.

## 레거시 정리

쿠키로 전환한 뒤에도 예상치 못한 곳에서 문제가 터졌다. 새로운 인증 구조를 만드는 것보다, 기존 구조의 흔적을 제거하는 작업이 실제로 더 오래 걸렸다.

### 산재한 localStorage 로직

프로젝트 전반에 토큰을 갱신하고 저장하는 로직이 흩어져 있었다. 쿠키로 전환했음에도 어딘가에서 여전히 localStorage에 토큰을 저장하는 코드가 남아 있었고, 또 다른 곳에서는 localStorage에서 토큰을 읽으려 시도하고 있었다.

문제는 이런 코드가 한 곳에 모여 있지 않았다는 것이다. 토큰을 다루는 유틸 함수가 여러 벌 존재했고, 어떤 컴포넌트는 유틸을 쓰고 어떤 컴포넌트는 `localStorage.getItem()`을 직접 호출했다. 쿠키 전환 후에도 이런 코드가 남아 있으면, 쿠키에는 새 토큰이 있는데 localStorage에는 옛 토큰이 남아 있는 상태가 된다. 결국 어떤 요청은 유효한 토큰으로, 어떤 요청은 만료된 토큰으로 보내지는 상황이 발생한다.

프로젝트 전체에서 `localStorage`를 참조하는 코드를 검색하고, 토큰 관련 읽기·쓰기를 모두 쿠키 기반으로 교체하거나 제거했다. 동시에 중복된 토큰 유틸 함수들을 하나로 통합했다.

### 분산된 인증 검사 로직

쿠키에 토큰이 정상적으로 저장되어 있는데도 로그인 페이지로 리다이렉트되는 문제가 발생했다. 원인은 미들웨어 외에도 개별 페이지와 컴포넌트에서 토큰을 직접 검사하고 리다이렉트하는 로직이 따로따로 구현되어 있었기 때문이다.

예를 들어, 어떤 페이지는 `useEffect`에서 localStorage의 토큰을 확인하고 없으면 로그인으로 보냈다. 쿠키에는 유효한 토큰이 있지만 localStorage는 비어 있으니, 로그인된 사용자가 페이지에 접근할 때마다 튕겨나갔다. 미들웨어에서 토큰 관리를 일원화한 의미가 없어지는 것이다.

이런 인증 검사 로직을 모두 찾아서, 미들웨어의 판단을 따르도록 통합했다. 개별 컴포넌트는 더 이상 토큰의 존재 여부를 직접 판단하지 않는다. 인증이 필요한 페이지의 접근 제어는 미들웨어가 담당하고, 컴포넌트는 미들웨어를 통과했다면 인증된 상태라고 신뢰하는 구조로 바꾸었다.

## 회고

개발 2일, 사내 구성원 대상 테스트를 포함해 **총 1주일**이 소요되었다.

- **매 페이지 토큰 갱신 오버헤드 제거**: `getCurrentUser()` 호출마다 발생하던 불필요한 토큰 갱신 요청이 사라졌다.
- **간헐적 로그인 풀림 해결**: 토큰 만료 시간이 적절하게 재설정되었고, 미들웨어가 만료 시 자동으로 갱신한다.
- **토큰 저장소 일원화**: DB와 localStorage에 흩어져 있던 토큰이 쿠키 한 곳으로 통합되어, 상태 불일치 문제가 원천적으로 사라졌다.

전환 후 단 한 건의 로그인 실패 리포트도 없었다. 가장 성공적인 마이그레이션은 사용자가 아무것도 눈치채지 못하는 마이그레이션이다.

### 한계점

미들웨어에서 토큰을 갱신하는 방식에는 잠재적 한계가 있다. 예를 들어 동시에 여러 요청이 들어왔을 때, 각 요청의 미들웨어가 독립적으로 refresh token을 사용하여 갱신을 시도하면 race condition이 발생할 수 있다. 또한 미들웨어에서 외부 OAuth 서버 호출이 실패했을 때의 폴백 전략도 별도로 구현하지 않았다. 현재는 갱신에 실패하면 만료된 토큰이 그대로 전달되고, 이후 인증이 필요한 로직에서 실패 처리를 한다.

우리 서비스 규모에서는 아직 이런 문제가 실제로 발생하지 않았지만, 트래픽이 늘어나면 대응이 필요한 영역이다.
