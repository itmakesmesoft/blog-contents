여러 명이 동시에 개발하면서 가장 신경 쓰이는 순간 중 하나는 배포할 때다. 내 코드가 준비됐는데 다른 사람의 미완성 코드가 섞여 나갈 수 있다는 불안. 우리 팀도 그랬다.

프론트엔드 개발자 6명이 하나의 모노레포에서 작업하고 있었다. 각자 다른 기능을 만들면서 `develop`에 머지하고, 배포할 때는 `release` 브랜치로 체리픽하는 구조. 언뜻 보면 합리적이었다. 근데 체리픽 과정에서 커밋 순서가 꼬이면서 충돌이 터졌고, 그 충돌을 해결하다가 미완성 코드가 프로덕션에 딸려 올라간 적이 있었다. 체리픽이 위험하다는 건 다들 알고 있었지만, "지금까지 잘 됐으니까"로 넘기고 있었다. 한 번 터지고 나서야 제대로 해결하기로 했다.

---

## 기존 배포 방식

우리 모노레포에는 서브 프로젝트 여러 개가 들어 있고, 6명이 각자 다른 기능을 개발한다. 작업이 끝나면 `develop` 브랜치에 머지하고, 배포 시점이 되면 배포할 커밋만 골라서 `release` 브랜치에 체리픽하는 방식이었다. 미완성 기능은 체리픽하지 않으면 프로덕션에 올라가지 않으니까, 완성된 것만 골라 배포할 수 있다는 게 이 구조의 전제였다.

---

## 체리픽의 문제

그 전제는 서브 프로젝트들이 서로 독립적일 때만 성립한다. 현실은 달랐다.

공유 컴포넌트를 수정하거나 라우팅 구조를 건드리면 다른 사람의 코드에 영향이 간다. 이런 상황에서 특정 커밋만 골라서 체리픽하면, 의존하는 코드는 빠지고 의존받는 코드만 올라가는 식의 불완전한 상태가 만들어진다. 충돌이 나면 더 복잡해진다. 충돌을 해결하는 과정에서 어떤 코드가 딸려 들어갔는지 확인하려면 커밋 하나하나를 대조해야 했고, 그 부담은 고스란히 배포 담당자에게 돌아갔다.

서브 프로젝트를 독립적으로 분리하면 좋겠지만, 현실적으로 공유하는 코드가 너무 많았다. 체리픽 자체를 없앨 방법이 필요했다.

---

## Feature Flag

팀 슬랙에서 이 문제를 논의하다가 Feature Flag라는 개념을 알게 됐다. 블로그 글들을 찾아보니 기능별로 토글을 만들고 DB와 연동해서, 특정 기능을 사용자에게 보여줄지 말지를 런타임에 제어하는 방식이 많았다.

근데 우리한테 필요한 건 그렇게 복잡하지 않았다. `develop` 환경에서는 모든 기능이 보이고, `production`에서는 완성된 기능만 보이면 됐다. 환경변수 하나로 현재 환경을 구분하고, 미완성 기능에 조건문만 달면 충분했다. DB도 필요 없고, Vercel이나 GitHub Actions에서 환경변수만 바꿔주면 되니까 추가 비용도 없었다.

이 방식이 체리픽 문제를 어떻게 해결하는지 정리하면 이렇다.

내가 음악 탭을 절반밖에 못 만들었어도, 다른 팀원이 채팅 버그를 고쳐서 배포해야 하면 `develop`을 그대로 `release`로 올리면 된다. 음악 탭 코드는 포함되지만 프로덕션에서는 숨겨져 있으니까 사용자한테 안 보인다. 기능이 완성되면 조건문만 제거하면 끝. 특정 커밋을 골라 담을 필요가 없으니 체리픽이 사라진다.

---

## 구현

`NEXT_PUBLIC_ENV` 환경변수 하나로 현재 환경을 구분하고, 그 위에 유틸리티 몇 개를 얹었다. 전체 코드가 파일 하나, 수십 줄이다.

**환경 판별 함수**

```tsx
// feature-flag.tsx

type EnvType = "development" | "production";

const currentEnv = (process.env.NEXT_PUBLIC_ENV ?? "development") as EnvType;

export const isEnv = (env: EnvType | EnvType[]) => {
  const envs = Array.isArray(env) ? env : [env];
  return envs.includes(currentEnv);
};

export const isDevelopment = isEnv("development");
export const isProduction = isEnv("production");
```

`isEnv()`는 현재 환경이 특정 환경인지 확인한다. 배열도 받을 수 있어서 `isEnv(["development", "staging"])` 같은 표현도 된다. `isDevelopment`와 `isProduction`은 자주 쓰니까 미리 계산해 뒀다.

**특정 환경에서만 함수 실행**

```tsx
export const onlyInEnv = (env: EnvType | EnvType[]) => {
  return <T extends (...args: any[]) => any>(fn: T): T => {
    if (isEnv(env)) return fn;
    return (() => {}) as T;
  };
};
```

**JSX 조건부 렌더링**

```tsx
export const OnlyInEnv = ({
  targetEnv,
  children,
}: {
  targetEnv: EnvType | EnvType[];
  children: React.ReactNode;
}) => {
  if (!isEnv(targetEnv)) return null;
  return <>{children}</>;
};
```

## 사용 패턴

가장 많이 쓴 패턴은 탭이나 메뉴를 숨기는 것이었다. 아직 개발 중인 음악/오디오 탭, 리워드 카드 같은 기능은 프로덕션에서 보이면 안 됐다. 네비게이션 설정에 `hidden: isProduction`만 추가하면 끝.

```tsx
const tabs = [
  { label: "채팅", path: "/chat" },
  { label: "음악", path: "/music", hidden: isProduction },
  { label: "리워드", path: "/reward", hidden: isProduction },
];
```

화면 단위로 통째로 숨겨야 할 때는 `<OnlyInEnv>`로 감쌌다.

```tsx
<OnlyInEnv targetEnv={["development"]}>
  <BetaFeature />
</OnlyInEnv>
```

외부 서비스 연동에도 유용했다. 개발 환경에서는 디버그 로거만 돌리고, 프로덕션에서는 Mixpanel, FullStory 같은 서비스를 초기화하는 식이다.

```tsx
const initAnalytics = onlyInEnv("production")(() => {
  Mixpanel.init(MIXPANEL_TOKEN);
  FullStory.init(FULLSTORY_ORG_ID);
});

const initDebugLogger = onlyInEnv("development")(() => {
  console.log("[Debug] Analytics disabled in development");
});
```

도입 이후 10개 넘는 파일에서 이 유틸리티를 쓰게 됐다. 각자 자기 기능에 플래그를 붙이고, 완성되면 제거하는 흐름이 자연스럽게 자리잡았다. 다만 플래그 제거를 강제하는 린트 규칙이나 체크리스트 같은 건 아직 없다. 지금까지는 파일 수가 적어서 문제가 안 됐지만, 늘어나면 관리 방법을 따로 만들어야 할 것 같다.

---

## 마무리

그 뒤로 체리픽이 사라졌다. `develop`에서 작업하고, 준비되면 그대로 `release`로 머지한다. 미완성 기능은 플래그가 숨겨준다. 환경변수 하나, 유틸 함수 몇 개로 6명이 서로 발목 안 잡고 배포할 수 있게 됐다.

해결책 자체는 별거 아니었다. 오히려 왜 진작 안 했나 싶다. 체리픽할 때마다 "빠뜨린 거 없나" 확인하느라 쓰던 시간, 배포 담당자한테 몰리던 부담. 결국 한 번 터져야 움직이게 되더라.
