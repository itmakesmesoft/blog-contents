AI 캐릭터 채팅앱의 크리에이터용 스튜디오를 만들고 있다. 크리에이터가 캐릭터를 만들고, 스토리를 작성하고, 이미지와 음악을 생성하는 도구다.

처음에는 이미지 생성 하나였다. 프롬프트를 입력하면 AI가 캐릭터 이미지를 만들어주는 기능. 그런데 오디오가 추가되었고, 비디오가 추가되었다. 각각 별도 모듈로 구현되었다. `ImageTask`, `AudioTask`, `VideoTask` — 서브콜렉션도, 클라우드 함수도, 클라이언트 훅도 전부 따로였다.

문제는 세 모듈이 80%의 로직을 공유하고 있다는 점이었다. 요청을 받아 Firestore 문서를 생성하고, 상태를 `pending` → `processing` → `completed`로 전환하고, 실패하면 에러를 기록하고, 완료되면 포인트를 차감하고, 결과물을 Storage에 저장한다. 이 흐름이 세 모듈에서 거의 동일하게 반복되고 있었다.

세 모듈을 합치되, 수 분짜리 비동기 작업과 실시간 상태 추적은 유지해야 했다.

## 전체 구조: 요청과 처리 분리하기

### 초기 설계: onCreate → Pub/Sub → Cloud Run Function

초기 아키텍처는 구성 요소가 많았다. `AssetGeneration` 문서가 생성되면 `onCreate` 트리거가 Pub/Sub 메시지를 발행하고, Cloud Run Function이 그 메시지를 받아 AI API를 호출하는 구조였다. 에셋 타입별로 `AssetGeneration/{generationId}/ImageTask`, `VideoTask` 같은 서브콜렉션으로 분리했다.

Cloud Run Function이 진행 상태를 Firestore에 업데이트하면 `onUpdate` 트리거가 후처리를 담당했다. 생성된 결과물은 Cloud Storage에 저장되었다.

Pub/Sub 메시지 큐가 재시도를 자동 처리해줬고, Cloud Run은 메모리와 타임아웃을 유연하게 조절할 수 있었다. 대신 구성 요소가 많아 디버깅이 복잡했고, Pub/Sub 메시지 지연이 간헐적으로 발생했다. 서브콜렉션 관리도 번거로웠다.

### 개선: onCreate에서 직접 처리

Cloud Functions v2가 540초 타임아웃과 2GB 메모리를 지원하면서, Pub/Sub 중간 계층을 제거할 수 있게 되었다. `ImageTask`/`VideoTask` 서브콜렉션도 단일 `AssetGeneration` 콜렉션에 `assetType` 필드를 추가하는 것으로 단순화했다.

요청 함수는 가볍다. 검증하고, Firestore 문서를 생성하고, `generationId`를 즉시 반환한다.

```typescript
// CreateAssetGeneration — onCall (60초 타임아웃, 512MB)
```

처리 함수가 실제 AI 생성을 담당한다. `pending` 상태를 확인하고, `processing`으로 전환한 뒤, `assetType`에 따라 분기한다.

```typescript
    } else if (generation.assetType === "audio") {
      // 오디오 생성 → Storage 업로드
    } else {
      // 비디오 생성 → Storage 업로드
    }

    // 트랜잭션: 완료 상태 업데이트 + 포인트 차감 + 에셋 생성
    await db.runTransaction(async (transaction) => {
```

요청과 처리를 분리하면 처리 함수가 크래시해도 `pending` 문서가 Firestore에 남아 있어서 정리할 수 있다. 타임아웃도 요청은 60초, 처리는 540초로 나눌 수 있고, 클라이언트는 `generationId`를 받는 즉시 실시간 구독을 시작할 수 있다. Firestore 문서가 두 함수 사이의 계약서 역할을 한다.

Pub/Sub을 빼고 나니 오히려 안정적이었다. 중간 계층이 많다고 안정성이 높아지는 건 아니다.

## 비디오 생성

이미지와 오디오는 비교적 단순하다. API 한 번 호출하면 결과가 나온다. 비디오는 달랐다. 4단계 파이프라인을 거쳐야 했고, 각 단계에서 실패할 수 있었다.

### 이미지 확장 (Outpainting)

사용자의 캐릭터 이미지는 1:1 비율이다. 비디오는 16:9가 필요하다. 단순히 이미지를 늘리면 컨텍스트가 손실되어 비디오 품질이 떨어진다.

Stability AI의 아웃페인팅(Outpainting) API를 사용했다. 원본 이미지의 비율과 목표 비율을 비교해서 패딩을 계산하고, AI가 자연스러운 배경을 채워 넣는다.

```typescript
export async function outpaintImageToAspectRatio(params: {
  imageBase64: string;
  imageMimeType: string;
  aspectRatio: string;
  additionalInfo?: string;
}): Promise<{ base64: string; mimeType: string }> {
  const imageBuffer = Buffer.from(imageBase64, "base64");
  const metadata = await sharp(imageBuffer).metadata();

  const [ratioW, ratioH] = aspectRatio.split(":").map(Number);
  const targetRatio = ratioW / ratioH;
  const currentRatio = imageWidth / imageHeight;

  // 이미 목표 비율이면 건너뛰기
  if (Math.abs(currentRatio - targetRatio) < 0.01) {
    return { base64: imageBase64, mimeType: imageMimeType };
  }

  // 패딩 계산: 넓으면 세로 확장, 좁으면 가로 확장
  let newWidth: number, newHeight: number;
  if (currentRatio >= targetRatio) {
    newWidth = imageWidth;
    newHeight = Math.ceil(imageWidth / targetRatio);
  } else {
    newHeight = imageHeight;
    newWidth = Math.ceil(imageHeight * targetRatio);
  }

  const left = Math.floor((newWidth - imageWidth) / 2);
  const right = newWidth - imageWidth - left;
  const up = Math.floor((newHeight - imageHeight) / 2);
  const down = newHeight - imageHeight - up;

  // Stability AI API 호출
  const payload = new FormData();
  payload.append("image", imageBuffer, { filename: "image.png" });
  payload.append("left", left.toString());
  payload.append("right", right.toString());
  payload.append("up", up.toString());
  payload.append("down", down.toString());
  payload.append(
    "prompt",
    `Extend the image with a natural and coherent background. ${
      additionalInfo ? `(Additional information: ${additionalInfo})` : ""
    }`,
  );

  const response = await fetch(
    "https://api.stability.ai/v2beta/stable-image/edit/outpaint",
    { method: "POST", headers, body: new Uint8Array(payload.getBuffer()) },
  );

  // ...
}
```

`additionalInfo`에 게임 메타데이터(장르, 분위기 등)를 함께 전달하면 확장된 배경이 캐릭터 세계관에 맞게 채워진다.

### 프롬프트 최적화

사용자는 한국어로 프롬프트를 입력한다. 하지만 Veo는 영어 프롬프트에서 최적의 결과를 낸다. 단순 번역이 아니라, 이미지의 시각적 스타일까지 분석해야 했다. 2D 애니메이션 이미지에 실사 스타일의 동영상이 생성되면 결과물이 어색해진다.

Gemini에게 이미지와 프롬프트를 함께 전달하고, 5-Part Formula에 따라 최적화된 영어 프롬프트를 생성하도록 했다.

### Veo 비디오 생성과 폴링

비디오 생성은 1~8분이 걸린다. Cloud Function 내부에서 동기 폴링(Polling)으로 완료를 기다린다.

```typescript
class VeoProviderService {
  async generateVideo(args: {
    model: VideoModel;
    prompt: string;
    imageBase64: string;
    imageMimeType: string;
    config?: Record<string, unknown>;
  }): Promise<VideoGenerationResult> {
    let operation = await genAI.models.generateVideos({
      model,
      prompt,
      image: { imageBytes: imageBase64, mimeType: imageMimeType },
      config,
    });

    // 540초 타임아웃 내에서 폴링 (최대 480회)
    const maxAttempts = 480;
    let attempts = 0;

    while (!operation.done && attempts < maxAttempts) {
      await this.delay(1000);
      attempts++;

      operation = await genAI.operations.getVideosOperation({ operation });

      if (attempts % 30 === 0) {
        functions.logger.info(
          `[VeoProvider] Still processing... (${attempts}s elapsed)`,
        );
      }
    }

    // RAI 필터링 확인
    if (operation.response?.raiMediaFilteredCount) {
      return { error: { message: "AI 안전 필터에 의해 차단되었습니다" } };
    }

    const video = operation.response?.generatedVideos?.[0]?.video;

    // videoBytes 우선, 없으면 URI에서 다운로드
    if (video.videoBytes) {
      return { videoData: video.videoBytes, mimeType: video.mimeType };
    }

    if (video.uri) {
      const response = await fetch(`${video.uri}?key=${apiKey}`);
      const buffer = Buffer.from(await response.arrayBuffer());
      return { videoData: buffer.toString("base64"), mimeType: "video/mp4" };
    }
  }
}
```

`maxAttempts` 480은 540초 타임아웃에서 여유분을 뺀 값이다. 30초마다 로그를 찍어서 어디까지 진행됐는지 확인하고, RAI(Responsible AI) 필터에 걸린 요청은 에러 메시지와 함께 실패 처리한다.

`videoBytes`를 우선 쓰고, 없으면 `uri`에서 다운로드하는 폴백도 있다. Veo API 응답 형태가 일관되지 않아서 넣은 방어 로직이다.

`VideoGenerationService.generateVideo`는 4단계를 순서대로 실행한다.

## 클라이언트: 수 분의 대기를 자연스럽게 만들기

서버에서 비디오가 만들어지는 수 분 동안 클라이언트 쪽에서 처리할 게 몇 가지 있다.

### 모듈 레벨 Firestore 구독

사이드패널을 닫거나 페이지를 이동해도 구독이 유지되어야 한다. React 컴포넌트의 라이프사이클에 구독을 묶으면, 언마운트 시 구독이 해제되어 상태 업데이트를 놓치게 된다.

구독을 React 바깥, 모듈 레벨로 올렸다.

```typescript
import { create } from "zustand";
import { doc, onSnapshot, Unsubscribe } from "firebase/firestore";

// React 바깥 — 컴포넌트 언마운트 영향 없음
const subscriptions = new Map<string, Unsubscribe>();

const cleanupSubscription = (generationId: string) => {
  const unsub = subscriptions.get(generationId);
  if (unsub) {
    unsub();
    subscriptions.delete(generationId);
  }
};

export const useGenerationTaskStore = create((set, get) => ({
  tasks: {}, // generationId → task

  track: (uid, generationId) => {
    if (subscriptions.has(generationId)) return;

    const docRef = doc(
      firestore,
      "Users",
      uid,
      "AssetGeneration",
      generationId,
    );

    const unsub = onSnapshot(docRef, (snapshot) => {
      if (!snapshot.exists()) return;

      const task = { ...snapshot.data(), id: snapshot.id };

      set((state) => ({
        tasks: { ...state.tasks, [generationId]: task },
      }));

      // 터미널 상태면 구독 자동 해제
      if (task.status === "completed" || task.status === "failed") {
        cleanupSubscription(generationId);
      }
    });

    subscriptions.set(generationId, unsub);
  },

  cleanup: () => {
    subscriptions.forEach((unsub) => unsub());
    subscriptions.clear();
    set({ tasks: {} });
  },
}));
```

`subscriptions` Map이 모듈 스코프에 있으므로 React 렌더 사이클과 무관하게 유지된다. `track`을 호출하면 Firestore `onSnapshot` 구독이 시작되고, 서버에서 상태가 변경될 때마다 Zustand 스토어가 갱신된다. 터미널 상태(`completed` 또는 `failed`)에 도달하면 구독이 자동으로 해제된다.

React 컴포넌트 바깥에서 구독을 관리하면, 구독 해제 시점은 누가 결정하는가. 답은 서버다. 서버가 터미널 상태를 기록하면 클라이언트가 이를 감지하고 스스로 정리한다.

### 낙관적 업데이트 — 5개 캐시 동시 갱신

생성 요청이 성공하면, 서버 응답을 기다리지 않고 즉시 UI를 업데이트한다. `useGenerateVideoMutation`의 `onSuccess`에서 5개의 React Query 캐시를 동시에 갱신한다.

```typescript
export const useGenerateVideoMutation = () => {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: async (params) => {
      const result = await firebaseClient.callFunction<GenerateVideoResult>(
        "CreateAssetGeneration",
        { ...params, platform: "studio" }
      )
      return { id: result.generationId, status: "pending", ... }
    },

    onSuccess: (data) => {
      // 1. placeholder VideoAsset 생성 (videoUrl = "" = 생성 중)
      const newVideoAsset = {
        id: data.id,
        assetType: "video",
        videoUrl: "", // 빈 문자열이 "생성 중" 표시 트리거
        metadata: { origin: "ai-generated", generationId: data.id },
      }

      // 2. "video" 필터 캐시 — 첫 번째 페이지에 삽입
      queryClient.setQueryData(["ai-assets", "video", "infinite"], (old) => ({
        ...old,
        pages: old.pages.map((page, i) =>
          i === 0 ? { ...page, data: [newVideoAsset, ...page.data] } : page
        ),
      }))

      // 3. "all" 필터 캐시 — 동일하게 삽입
      queryClient.setQueryData(["ai-assets", "all", "infinite"], ...)

      // 4. latest-generation 캐시 (InProgressBanner 감지용)
      queryClient.setQueryData([LATEST_GENERATION_QUERY_KEY, "video"], data)

      // 5. Video model counts 낙관적 차감
      queryClient.setQueryData([VIDEO_MODEL_COUNTS_QUERY_KEY], (old) => ({
        ...old,
        [data.model]: {
          ...old[data.model],
          used: old[data.model].used + 1,
          remaining: Math.max(old[data.model].remaining - 1, 0),
        },
      }))
    },
  })
}
```

`generationId`가 전체 파이프라인을 관통하는 상관관계 키(Correlation Key)다. 요청 시점에 생성된 이 ID가 서버의 Firestore 문서, 클라이언트의 Zustand 구독, React Query 캐시의 placeholder, 완료 시 교체 로직까지 하나로 연결한다.

### 완료/실패 시 캐시 교체

`useGenerationTaskEffects`가 Zustand 스토어의 변경을 감지하고, 터미널 상태에 도달하면 React Query 캐시를 교체한다.

```typescript
export const useGenerationTaskEffects = () => {
  const queryClient = useQueryClient();
  const { showToast } = useToast();

  useEffect(() => {
    const unsub = useGenerationTaskStore.subscribe((state, prevState) => {
      for (const [id, task] of Object.entries(state.tasks)) {
        const prev = prevState.tasks[id];
        if (!prev || prev.status === task.status) continue;

        // 완료: generationId로 placeholder를 실제 데이터로 교체
        if (task.status === "completed") {
          queryClient.setQueryData(
            ["ai-assets", "video", "infinite"],
            (old) => ({
              ...old,
              pages: old.pages.map((page) => ({
                ...page,
                data: page.data.map((asset) =>
                  asset.metadata?.generationId === id
                    ? { ...asset, videoUrl: task.videoUrl }
                    : asset,
                ),
              })),
            }),
          );
        }

        // 실패: 캐시에서 placeholder 제거 + 에러 토스트
        if (task.status === "failed") {
          queryClient.setQueryData(
            ["ai-assets", "video", "infinite"],
            (old) => ({
              ...old,
              pages: old.pages.map((page) => ({
                ...page,
                data: page.data.filter(
                  (asset) => asset.metadata?.generationId !== id,
                ),
              })),
            }),
          );
          showToast({ type: "error", message: "비디오 생성에 실패했습니다" });
        }
      }
    });
    return unsub;
  }, [queryClient, showToast]);
};
```

완료 시에는 `generationId`로 placeholder를 찾아 `videoUrl`을 실제 URL로 교체한다. 실패 시에는 캐시에서 placeholder를 제거하고 에러 토스트를 표시한다. 사용자 입장에서는 비디오 카드가 로딩 상태에서 완성된 비디오로 자연스럽게 전환되거나, 실패 시 카드가 사라지면서 알림을 받는 경험이다.

## 안전장치: 멈춘 작업 자동 정리

Cloud Function이 크래시하거나, 외부 AI API가 무응답 상태에 빠지면 작업이 `pending` 또는 `processing` 상태에서 멈출 수 있다. 이 상태로 방치되면 클라이언트는 영원히 로딩을 표시하게 된다.

5분마다 실행되는 스케줄러가 이를 정리한다.

```typescript
export const CleanUpStuckAssetGenerationsOnSchedule = functions.pubsub
  .schedule("every 5 minutes")
  .onRun(async () => {
    const tenMinutesAgo = new Date(Date.now() - 10 * 60 * 1000);
    const db = admin.firestore();

    // 전체 사용자 대상 collectionGroup 쿼리
    const stuckTasks = await db
      .collectionGroup("AssetGeneration")
      .where("status", "in", ["pending", "processing"])
      .where("updatedAt", "<", tenMinutesAgo)
      .orderBy("updatedAt", "asc")
      .limit(100)
      .get();

    if (stuckTasks.empty) return;

    const batch = db.batch();
    for (const taskDoc of stuckTasks.docs) {
      batch.update(taskDoc.ref, {
        status: "failed",
        error: "Timeout - automatic failure",
        updatedAt: FieldValue.serverTimestamp(),
      });
    }
    await batch.commit();
  });
```

10분 임계값은 비디오 생성(최대 8분) + 여유를 고려한 값이다. `collectionGroup` 쿼리로 전체 사용자의 `AssetGeneration`을 한 번에 조회하고, `batch` 처리로 한꺼번에 업데이트한다. `failed` 상태로 전환되면 클라이언트의 `onSnapshot` 구독이 이를 감지하고, placeholder를 제거하고 에러 토스트를 표시한다.

이 스케줄러가 없으면 어떤 일이 벌어질까. 클라이언트는 영원히 로딩 스피너를 돌리고, 사용자는 브라우저를 새로고침하고, 그래도 로딩 중이다.

## 포인트와 할당량: 성공 후 차감 패턴

에셋 생성에는 포인트가 필요하다. 차감 시점에 두 가지 선택지가 있었다.

| 방식          | 장점                   | 단점                     |
| ------------- | ---------------------- | ------------------------ |
| 선차감 후환급 | 동시 생성 방지 용이    | 환급 실패 시 포인트 손실 |
| 성공 후 차감  | 포인트 안전, 로직 단순 | 동시 생성 별도 방어 필요 |

성공 후 차감을 택했다. 환급이 실패하면 포인트가 영구적으로 날아가는데, 이건 사용자 신뢰를 깨뜨린다. Cloud Functions에서 환급 로직을 추가하면 실패 경로도 복잡해진다.

포인트 차감은 생성 완료와 같은 트랜잭션(Transaction) 안에서 처리된다.

```typescript
// AssetGenerationOnCreate 내부
await db.runTransaction(async (transaction) => {
  // 1. 포인트 데이터 읽기
  const [prevUserPoint, remainingTransactions] = await Promise.all([
    userPointsService.findByInTransaction(transaction, { uid: userId }),
    userPointTransactionsService.findRemainingAmountTransactions({
      uid: userId, transaction,
    }),
  ]);

  // 2. 생성 완료 상태 업데이트
  transaction.update(generationRef, {
    status: "completed",
    ...updateData,
    completedAt: FieldValue.serverTimestamp(),
  });

  // 3. 포인트 차감 (같은 트랜잭션)
  if (needsPointDeduction && prevUserPoint) {
    userPointsService.withdrawPointsInTransaction(transaction, {
      uid: userId,
      amount: generation.pointCost,
      prevUserPoint,
      remainingExpirablePointTransactions,
    });
  }

  // 4. 에셋 자동 저장 (같은 트랜잭션)
  if (generation.autoSaveToAssets) {
    const assetRef = db.collection("Users").doc(userId)
      .collection("Assets").doc();
    transaction.set(assetRef, buildVideoAssetData(...));
    transaction.update(generationRef, { assetId: assetRef.id });
  }
});
```

생성 완료, 포인트 차감, 에셋 저장이 하나의 트랜잭션에서 원자적(Atomic)으로 처리된다. 포인트 차감이 실패하면 생성 완료도 롤백된다. 부분적으로 완료된 상태가 존재할 수 없다.

일일 할당량(Daily Quota)은 `YYYY-MM-DD` 형식의 문서 ID로 관리한다. 별도의 리셋 로직이 필요 없다. 날짜가 바뀌면 새 문서가 생성되고, 이전 날짜의 문서는 자연스럽게 의미를 잃는다.

## 페이로드 제한과 이미지 압축

이미지를 Base64로 인코딩하면 33%의 오버헤드가 발생한다. 10MB 이미지가 13MB+가 된다. Cloud Functions의 onCall 페이로드 제한은 10MB다.

클라이언트에서 WebP 압축과 최대 1920px 리사이즈를 적용해서 전송 크기를 줄였다. 하지만 비디오 생성에서는 이것만으로 부족했다.

비디오 생성의 해결책은 이미지를 onCall에서 Cloud Storage에 먼저 업로드하고, onCreate에서 URL로 fetch하는 것이었다.

```typescript
// videoGenerationService.createRequest 내부
const imageUrl = await storageProviderService.uploadImageFromBase64({
  generationId: validatedData.generationId || db.collection("_").doc().id,
  imageData: validatedData.imageBase64,
  mimeType: validatedData.imageMimeType,
  subfolder: "imageUrl",
});
```

onCall 함수는 이미지 바이너리를 Firestore 문서에 저장하는 대신 Storage에 업로드하고, URL만 문서에 기록한다. onCreate 함수는 이 URL에서 이미지를 fetch한다. Firestore 문서 크기 제한(1MB)도 피하고, 페이로드 문제도 해결한다.

## 무엇이 의미 있는 지표인가

비디오 생성 기능을 출시한 뒤, 사용자 3명에게 설문을 돌렸다. "생성한 비디오를 실제 게임에 적용했나요." 답은 0/3. 아무도 적용하지 않았다.

처음에는 당황스러웠다. 하지만 곰곰이 생각해보니, 핵심 액션(Key Action)을 잘못 정의하고 있었다. "비디오 적용"이 아니라 "비디오 생성" 자체가 이 기능의 가치였다. 크리에이터들은 캐릭터가 움직이는 모습을 보는 것 자체에서 만족감을 느꼈다. 트레일러로 적용하는 것은 부가적인 행위였다.

리텐션(Retention) 지표도 재정의해야 했다. Daily Retention은 의미가 없었다. 매일 트레일러를 만드는 크리에이터는 없다. Weekly 또는 Monthly Active로 봐야 의미 있는 패턴이 보였다.

분석 도구도 하나로는 부족했다.

| 도구        | 용도             | 발견                    |
| ----------- | ---------------- | ----------------------- |
| Mixpanel    | 정량적 이벤트    | 생성 요청 수, 적용 건수 |
| FullStory   | 세션 리플레이    | 버튼 미클릭 패턴        |
| 설문조사    | 정성적 피드백    | 만족도, 불편사항        |
| ChannelTalk | 타겟 유저 인터뷰 | 적용하지 않는 이유      |

Mixpanel만 보면 "생성은 하지만 적용은 안 한다"까지만 알 수 있다. FullStory로 세션을 리플레이하면 "적용 버튼을 찾지 못하고 있다"가 보인다. 설문으로는 "비디오 품질이 기대에 못 미친다"는 피드백이 나온다. ChannelTalk 인터뷰에서는 "적용 방법을 모르겠다"는 답이 돌아온다. 하나만 봤으면 엉뚱한 데 시간을 썼을 거다.

## 마무리

돌아보면, 요청과 처리를 분리한 게 나머지 설계를 많이 결정지었다. `pending` 문서가 남으니까 cleanup이 되고, Firestore 구독으로 실시간 추적이 되고, 트랜잭션으로 과금이 묶인다. `generationId` 하나가 서버부터 클라이언트 캐시까지 전부 이어준다.

기술적으로는 만족스럽다. 그런데 적용률 0%를 보면서, 파이프라인이 잘 돌아가는 것과 사용자가 결과물을 쓰는 것은 다른 문제라는 걸 느꼈다. 코드 짜는 시간만큼 사용자 관찰에 시간을 쓸 걸 그랬다.
