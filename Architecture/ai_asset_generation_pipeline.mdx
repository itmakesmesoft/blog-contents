AI 캐릭터 채팅앱의 크리에이터 스튜디오를 만들고 있다. 이미지, 오디오, 영상을 AI로 생성하는 기능이 있는데, 세 모듈이 80%의 로직을 공유하면서도 서브콜렉션, 클라우드 함수, 클라이언트 훅이 전부 따로였다. 이걸 하나의 파이프라인으로 합치면서, 수 분짜리 비동기 처리와 실시간 Status 추적을 어떻게 풀었는지 정리한다.

## 요청과 처리 분리하기

### 초기 설계와 그 한계

  alt="image"
/>

초기 아키텍처는 구성 요소가 많았다. `AssetGeneration` 문서가 생성되면 `onCreate` 트리거가 Pub/Sub 메시지를 발행하고, Cloud Run Function이 그 메시지를 받아 AI API를 호출하는 구조였다. 에셋 타입별로 `AssetGeneration/{generationId}/ImageTask`, `VideoTask` 같은 서브콜렉션으로 분리했다. 이렇게 하나의 generationID 하위에 서브컬렉션으로 Task를 두면 하나의 생성 요청에서 멀티 프롬프팅을 하는 경우도 대응이 가능했다.

Pub/Sub이 재시도를 자동 처리해줬고, Cloud Run은 메모리와 타임아웃을 유연하게 조절할 수 있었다. 하지만 개발 과정에서 '굳이 이렇게 할 필요가 있을까?' 싶은 부분들이 보이기 시작했다.

일단 중간에 멀티 프롬프팅이 기획에서 사라졌는데, 지금의 설계로는 불필요하게 서브컬렉션으로 구분하면서 쿼리가 비효율적으로 요청되고 있었다. 이 경우에는 하나의 AssetGeneration 내에서 필드를 통해 타입만 구별하는게 훨씬 깔끔할 것 같았다. Pub/Sub 메시지가 간헐적으로 수 초씩 지연되면서 사용자가 "생성 중" 상태에서 오래 머무르는 경우가 있었다. 원인을 추적하려면 onCreate 트리거 로그, Pub/Sub 메시지 로그, Cloud Run 실행 로그를 세 군데에서 대조해야 했다. 구성 요소가 많다고 안정성이 높아지는 건 아니었다.

### Pub/Sub 대신 onCreate에서 생성 처리하기

일단 불필요한 Pub/Sub 중간 계층을 제거하고 onCreate에서 직접 생성을 요청하도록 구조를 바꾸어 파이프라인을 단순화하였다. `ImageTask`/`VideoTask` 서브콜렉션도 단일 `AssetGeneration` 콜렉션에 `assetType` 필드를 추가하는 것으로 단순화했다.

요청 함수는 가볍다. 검증하고, Firestore 문서를 생성하고, `generationId`를 즉시 반환한다.

```typescript
// CreateAssetGeneration — onCall (60초 타임아웃, 512MB)
export const CreateAssetGeneration = functions
  .region("asia-northeast3")
  .runWith({ timeoutSeconds: 60, memory: "512MB" })
  .https.onCall(async (data, context) => {
    const { uid } = await authGuard.canActivateCallable(context);
    const result = await assetGenerationService.requestGeneration({
      uid,
      data,
    });
    return result; // { generationId, status: "pending" }
  });
```

onCreate 함수가 저장된 명세를 토대로 실제 AI 생성을 수행한다. `pending` 상태를 확인하고, `processing`으로 전환한 뒤, `assetType`에 따라 분기한다.

```typescript
// AssetGenerationOnCreate — Firestore 트리거 (540초 타임아웃, 2GB)
export const AssetGenerationOnCreate = functions
  .region("asia-northeast3")
  .runWith({ timeoutSeconds: 540, memory: "2GB" })
  .firestore.document("Users/{userId}/AssetGeneration/{generationId}")
  .onCreate(async (snapshot, context) => {
    const { userId, generationId } = context.params;
    const generation = snapshot.data() as AssetGeneration;

    if (generation.status !== "pending") return;

    await snapshot.ref.update({
      status: "processing",
      updatedAt: FieldValue.serverTimestamp(),
    });

    // 공통 흐름: assetType에 따라 생성 로직만 분기
    const result = await generateByType(generation);
    // ├─ "image" → imageGenerationService.generate()
    // ├─ "audio" → audioGenerationService.generate()
    // └─ "video" → videoGenerationService.generate()

    // 공통 흐름: 완료 상태 + 포인트 차감 + 에셋 저장을 트랜잭션으로
    await db.runTransaction(async (transaction) => {
      transaction.update(generationRef, {
        status: "completed",
        ...result,
      });
      // 포인트 차감, 에셋 자동 저장 ...
    });
  });
```

세 모듈이 공유하던 80%의 로직 — Status 관리, 에러 처리, 포인트 차감, Storage 저장 — 이 `AssetGenerationOnCreate` 하나에 모인다. 다른 20%인 실제 AI API 호출만 `assetType`으로 분기한다.

요청과 처리를 분리하면 처리 함수가 비정상 종료되어도 `pending` 문서가 Firestore에 남아 있어서 나중에 확인할 수 있다. 즉 AssetGeneration 문서는 일종의 생성 계약서 역할을 한다.

## 전 처리 단계의 실패 허용하기

이미지와 오디오는 API 한 번 호출하면 결과가 나온다. 영상은 다르다. 아웃페인팅(이미지 비율 확장) → 프롬프트 최적화(한→영 변환 + 스타일 분석) → Veo 영상 생성 → 후처리까지 4단계 파이프라인을 거치고, 각 단계에서 실패할 수 있다.

이때 전처리 단계의 경우 실패했다고 전체 생성을 중단하면 사용자는 아무것도 받지 못해 사용자 경험이 떨어질거라 생각했다. 그래서 비록 전처리 단계가 실패해 품질이 다소 떨어지더라도 결과를 돌려주는 쪽이 낫다고 판단했다.

```typescript
class VideoGenerationService {
  async generateVideo(args) {
    // 1. Storage에서 이미지 fetch
    const imageBase64 = await fetchImageAsBase64(imageUrl);

    // 2. Outpaint (실패 시 원본 이미지 사용)
    let finalImage = imageBase64;
    try {
      const outpainted = await outpaintImageToAspectRatio({ ... });
      finalImage = outpainted.base64;
    } catch { /* 원본 유지 */ }

    // 3. 프롬프트 최적화 (실패 시 원본 프롬프트 사용)
    let finalPrompt = prompt;
    try {
      finalPrompt = await generateVideoPrompt(prompt, finalImage, ...);
    } catch { /* 원본 유지 */ }

    // 4. Veo 영상 생성 — 이것만 실패하면 전체 중단
    return await veoProviderService.generateVideo({
      prompt: finalPrompt,
      imageBase64: finalImage,
      ...
    });
  }
}
```

아웃페인팅이 실패하면 원본 이미지로, 프롬프트 최적화가 실패하면 원본 프롬프트로 진행한다. 최종 Veo 생성만이 실패 시 전체 파이프라인을 중단한다.

## 사용자의 체감 대기시간 줄이기

영상 생성이 1~8분 걸리는 동안 클라이언트에서는 사용자가 자연스럽게 기다릴 수 있도록 처리해야 했다.

### 모듈 레벨 Firestore 구독

영상 생성은 열거나 닫을 수 있는 사이드패널 내에서 작업이 가능한데, 이때 패널을 닫거나 페이지를 이동해도 영상 생성 상태의 구독이 유지되어야 한다. 처음에는 React `useEffect` 안에서 `onSnapshot`을 구독했는데, 사용자가 사이드패널을 닫으면 컴포넌트가 언마운트되면서 cleanup 함수가 구독을 해제했다. 3분 뒤 서버가 영상 생성을 완료해도 클라이언트는 이를 감지하지 못한다. 사이드패널을 다시 열어야만 완료 결과를 볼 수 있었다.

구독 해제가 컴포넌트 언마운트에 묶여 있는 게 문제였다. 구독을 React 바깥, 모듈 레벨로 올렸다.

```typescript
import { create } from "zustand";
import { doc, onSnapshot, Unsubscribe } from "firebase/firestore";

// React 바깥 — 컴포넌트 언마운트 영향 없음
const subscriptions = new Map<string, Unsubscribe>();

const cleanupSubscription = (generationId: string) => {
  const unsub = subscriptions.get(generationId);
  if (unsub) {
    unsub();
    subscriptions.delete(generationId);
  }
};

export const useGenerationTaskStore = create((set, get) => ({
  tasks: {},

  track: (uid, generationId) => {
    if (subscriptions.has(generationId)) return;

    const docRef = doc(
      firestore,
      "Users",
      uid,
      "AssetGeneration",
      generationId,
    );

    const unsub = onSnapshot(docRef, (snapshot) => {
      if (!snapshot.exists()) return;

      const task = { ...snapshot.data(), id: snapshot.id };
      set((state) => ({
        tasks: { ...state.tasks, [generationId]: task },
      }));

      // completed/failed면 구독 자동 해제
      if (task.status === "completed" || task.status === "failed") {
        cleanupSubscription(generationId);
      }
    });

    subscriptions.set(generationId, unsub);
  },

  cleanup: () => {
    subscriptions.forEach((unsub) => unsub());
    subscriptions.clear();
    set({ tasks: {} });
  },
}));
```

`subscriptions` Map이 모듈 스코프에 있으므로 React 렌더 사이클과 무관하게 유지된다. `track`을 호출하면 Firestore `onSnapshot` 구독이 시작되고, 서버에서 Status가 변경될 때마다 Zustand 스토어가 갱신된다. `onSnapshot` 콜백이 `status === "completed"` 또는 `"failed"`를 감지하면 `cleanupSubscription`을 호출하고, 그 전까지는 사이드패널을 닫든 페이지를 이동하든 구독이 유지된다.

### 낙관적 업데이트와 캐시 교체

생성 요청 후 서버가 영상을 완성할 때까지 1~8분이 걸린다. 그동안 에셋 목록에 아무 변화가 없으면 사용자는 요청이 접수된 건지 알 수 없다.

요청 응답으로 `generationId`를 받는 즉시 에셋 데이터를 만들어 React Query 캐시에 먼저 삽입했다. `id`, `assetType`, `metadata`는 서버 응답 기반의 실제 값이고, `videoUrl`만 빈 문자열이다. 컴포넌트는 빈 `videoUrl`을 보고 로딩 카드를 렌더링한다.

```typescript
onSuccess: (data) => {
  // videoUrl = "" → 컴포넌트가 로딩 카드를 렌더링
  const newVideoAsset = {
    id: data.id,
    assetType: "video",
    videoUrl: "",
    metadata: { origin: "ai-generated", generationId: data.id },
  };

  // 에셋 목록 캐시에 삽입
  queryClient.setQueryData(["ai-assets", "video", "infinite"], (old) => ({
    ...old,
    pages: old.pages.map((page, i) =>
      i === 0 ? { ...page, data: [newVideoAsset, ...page.data] } : page,
    ),
  }));

  // 모델별 사용량 차감
  queryClient.setQueryData([VIDEO_MODEL_COUNTS_QUERY_KEY], (old) => ({
    ...old,
    [data.model]: {
      ...old[data.model],
      used: old[data.model].used + 1,
      remaining: Math.max(old[data.model].remaining - 1, 0),
    },
  }));
};
```

요청 시점에 생성된 `generationId` 하나가 서버의 Firestore 문서, 클라이언트의 Zustand 구독, React Query 캐시의 낙관적 에셋 데이터를 전부 연결한다. Status가 `completed` 또는 `failed`로 바뀌면 Zustand 스토어가 캐시를 갱신한다. 완료 시에는 `generationId`로 해당 에셋을 찾아 실제 `videoUrl`을 채우고, 실패 시에는 캐시에서 에셋을 제거하고 에러 토스트를 표시한다.

```typescript
useEffect(() => {
  const unsub = useGenerationTaskStore.subscribe((state, prevState) => {
    for (const [id, task] of Object.entries(state.tasks)) {
      const prev = prevState.tasks[id];
      if (!prev || prev.status === task.status) continue;

      if (task.status === "completed") {
        // 낙관적 에셋 데이터의 videoUrl을 실제 URL로 교체
        queryClient.setQueryData(["ai-assets", "video", "infinite"], (old) => ({
          ...old,
          pages: old.pages.map((page) => ({
            ...page,
            data: page.data.map((asset) =>
              asset.metadata?.generationId === id
                ? { ...asset, videoUrl: task.videoUrl }
                : asset,
            ),
          })),
        }));
      }

      if (task.status === "failed") {
        // 낙관적 에셋 데이터 제거 + 에러 알림
        queryClient.setQueryData(["ai-assets", "video", "infinite"], (old) => ({
          ...old,
          pages: old.pages.map((page) => ({
            ...page,
            data: page.data.filter(
              (asset) => asset.metadata?.generationId !== id,
            ),
          })),
        }));
        showToast({ type: "error", message: "영상 생성에 실패했습니다" });
      }
    }
  });
  return unsub;
}, [queryClient, showToast]);
```

사용자가 보는 건 단순하다. 비디오 카드가 로딩에서 완성으로 바뀌거나, 실패하면 카드가 사라지면서 토스트가 뜬다.

## 멈춰있는 작업들 자동 정리하기

위 구조가 동작하려면 Status가 반드시 `completed` 또는 `failed`에 도달해야 한다. 그런데 Cloud Function이 비정상 종료되거나 외부 AI API가 응답하지 않으면, 작업이 `pending` 또는 `processing`에서 멈출 수 있다. 이 상태로 방치되면 클라이언트는 영원히 로딩을 표시하게 된다.

일정 시간이 지나도 끝나지 않는 작업은 실패로 간주하기로 했다. 영상 생성이 최대 8분이니, 10분이 지나도 `pending`이나 `processing`이면 뭔가 잘못된 것이다. 그래서 5분마다 주기적으로 이런 작업을 찾아 `failed`로 Status를 전환해주는 스케줄링 함수를 만들었다.

```typescript
export const CleanUpStuckAssetGenerationsOnSchedule = functions.pubsub
  .schedule("every 5 minutes")
  .onRun(async () => {
    const tenMinutesAgo = new Date(Date.now() - 10 * 60 * 1000);

    // 전체 사용자 대상 collectionGroup 쿼리
    const stuckTasks = await db
      .collectionGroup("AssetGeneration")
      .where("status", "in", ["pending", "processing"])
      .where("updatedAt", "<", tenMinutesAgo)
      .orderBy("updatedAt", "asc")
      .limit(100)
      .get();

    if (stuckTasks.empty) return;

    const batch = db.batch();
    for (const taskDoc of stuckTasks.docs) {
      batch.update(taskDoc.ref, {
        status: "failed",
        error: "Timeout - automatic failure",
        updatedAt: FieldValue.serverTimestamp(),
      });
    }
    await batch.commit();
  });
```

`collectionGroup` 쿼리로 전체 사용자의 `AssetGeneration`을 한 번에 조회하고, `batch`로 한꺼번에 `failed`로 업데이트한다. Status가 바뀌면 클라이언트의 `onSnapshot` 구독이 이를 감지하고, 캐시에서 해당 에셋을 제거하고 에러 토스트를 표시한다.

이 스케줄러가 없으면 사용자는 브라우저를 새로고침해도 로딩 중인 화면만 보게 된다.

## 포인트 선차감 후생성 vs 선생성 후차감

에셋 생성에는 포인트가 필요하다. 차감 시점을 두고 두 가지 선택지가 있었다.

처음에는 선차감 후생성 방식을 택했다. 동시 생성 방지에 유리하기 때문이다. 하지만 생성이 실패하면 포인트를 환급해야 했고, 포인트 관련 로직이 onCreate와 onUpdate 모두에 걸치면서 코드 복잡도가 예상보다 커졌다. 리드개발자의 조언도 같은 방향이었다. 분기가 많아지면 버그가 생길 가능성이 높아진다고.

선생성 후차감으로 전환했다. 동시 생성 방어는 클라이언트에서 처리하고, 포인트 차감은 생성이 성공한 뒤에만 실행한다. 생성 완료, 포인트 차감, 에셋 저장을 하나의 트랜잭션에서 원자적으로 처리한다.

```typescript
await db.runTransaction(async (transaction) => {
  const [prevUserPoint, remainingTransactions] = await Promise.all([
    userPointsService.findByInTransaction(transaction, { uid: userId }),
    userPointTransactionsService.findRemainingAmountTransactions({
      uid: userId,
      transaction,
    }),
  ]);

  // 생성 완료
  transaction.update(generationRef, {
    status: "completed",
    ...updateData,
    completedAt: FieldValue.serverTimestamp(),
  });

  // 포인트 차감 (같은 트랜잭션)
  if (needsPointDeduction && prevUserPoint) {
    userPointsService.withdrawPointsInTransaction(transaction, {
      uid: userId,
      amount: generation.pointCost,
      prevUserPoint,
      remainingExpirablePointTransactions,
    });
  }

  // 에셋 자동 저장 (같은 트랜잭션)
  if (generation.autoSaveToAssets) {
    const assetRef = db.collection("Users").doc(userId)
      .collection("Assets").doc();
    transaction.set(assetRef, buildVideoAssetData(...));
    transaction.update(generationRef, { assetId: assetRef.id });
  }
});
```

하나의 트랜잭션이므로 포인트 차감이 실패하면 생성 완료도 롤백된다. 부분적으로 완료된 상태가 존재할 수 없다.

## 마무리

Pub/Sub을 제거하면서 자동 재시도를 잃었다. cleanup 스케줄러가 멈춘 작업을 처리해주긴 하지만, 이건 재시도가 아니라 포기 선언이다. 사용자는 실패 토스트를 받고 직접 다시 생성 버튼을 눌러야 한다. `pending` 상태의 문서가 Firestore에 남아 있으니 재시도 큐를 붙일 여지는 있는데, 아직 실패율이 높지 않아서 우선순위를 미뤘다.

서버리스의 제약 안에서 만든 구조라 완벽하진 않지만, 세 모듈에 흩어져 있던 코드가 하나의 파이프라인으로 모이고, `generationId` 하나로 서버부터 클라이언트 캐시까지 추적할 수 있게 된 건 이전보다 나아진 부분이다.
